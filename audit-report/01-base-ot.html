emp-ot — base OT (public-key world)

The public-key bootstrap layer under emp-ot/base_ot/: the chosen-input 1-out-of-2 base OTs that seed every OT extension. Three are elliptic-curve schemes over emp-tool's P-256 ECGroup — CO (Chou–Orlandi "Simplest OT"), PVW (messy-mode framework OT), and CSW (Canetti–Sarkar–Wang "Blazing Fast" FSF-rOT, the default base OT of every extension) — plus a post-quantum peer, BMM (Badrinarayanan–Masny–Mukherjee) over ML-KEM-512 with a vendored Kyber subset. This report covers their protocol flows, trust models, the shared RO / point-validation substrate, the correlation breaker and CRS derivation, the constant-time posture, and the per-scheme assumptions.

1 · Scheme inventory, trust model & live-vs-legacy

Four base OTs share emp-ot/base_ot/. Three are header-only over emp-tool's P-256 ECGroup — co.h:14, pvw.h:25, csw.h:49 — and the fourth, BMM, is lattice-based: bmm.h:64 declares the class while the ML-KEM algebra lives in mlkem/bmm.cpp. All four implement the abstract emp::OT 1-of-2 interface (ot.h:9-13).

The trust model of each scheme is declared, not inferred, by whether it overrides is_malicious_secure():

SchemeFamily / paperAssumptionis_malicious_secure()Runtime abort?

PVWPeikert–Vaikuntanathan–Waters framework OT (2007/348)DDH (receiver) + statistical (sender)true (pvw.h:28)none — structural
CSWCanetti–Sarkar–Wang FSF-rOT (SFOT.tex)CDH in the observable-RO modeltrue (csw.h:51)yes — hash-equality check via error()

CO is semi-honest only — do not treat it as a malicious base OT

CO defines only send/recv (co.h:18-69) and no is_malicious_secure() override, so it inherits the base-class default false (ot.h:16). Its flow is the plain Chou–Orlandi key exchange + RO KDF over B/BA with no consistency check against a cheating peer. This is the library's conservative flagging (the cited paper argues ROM security), but the two code facts hold: no override, no active check. A malicious receiver's Bi is on-curve-validated but nothing binds it to a well-formed choice.

CSW is the sole trait that drives an extension's fast path. It is the only base OT overriding supports_deferred_check() to true (csw.h:59; base default false at ot.h:23), exposing the messy-core + deferred-extraction-check split the 3-round extension overlap relies on. ot_extension.h static_casts the base OT to CSW* only under that trait (ot_extension.h:154, ot_extension.h:46 holds the resulting csw_base_), so the cast is sound iff no other base OT ever sets the trait — currently guaranteed.

Live vs legacy. IKNPBaseOT = CSW (iknp.h:13), SoftSpoken and Ferret alias the same, and ot_extension.h includes base_ot/csw.h directly (ot_extension.h:9). CO / PVW / BMM appear only in test/test_base_ot.cpp and bench/bench_base_ot.cpp. Git history confirms the churn: 8c6d4d6 replaced the old PVWKyber with BMM over ML-KEM-512 and de-layered CSW (folding the shared sfrot_check.h/CSWBaseOT abstraction back into csw.h); a990756 dropped Naor–Pinkas (OTNP); d5aed11 raised the BMM correlation-breaker parameter to λ=256.

2 · Shared substrate — group, point validation, CRS, the RO

All three EC schemes hold a value-owned ECGroup G member (co.h:15, pvw.h:30, csw.h:61) defaulting to P-256 / NID_X9_62_prime256v1. Only P-256 is wired for hash-to-curve; any other curve errors at runtime. Three substrate facts are load-bearing for every scheme.

Every received point is on-curve + non-infinity validated

Point::from_bin (emp-tool runtime/crypto/ec.cpp:335) is the sole EC_POINT_oct2point caller — the only path external bytes reach a Point — and it unconditionally re-checks EC_POINT_is_on_curve (rejecting off-curve, ec.cpp:345) and EC_POINT_is_at_infinity (rejecting the identity, ec.cpp:347), both via fatal error() (so they survive NDEBUG). recv_pt always calls from_bin (io_channel.h:215), so CO's Bi/A, PVW's (gi,hi)/(u,c), and CSW's Bi/z are all validated on receipt.

Subgroup safety rides on P-256's cofactor being 1

The comment at ec.cpp:342 argues on-curve + non-identity implies prime-order-subgroup membership because P-256 has cofactor h=1. Correct for P-256 — and P-256 is the default and sole wired curve — but the invariant silently depends on the curve staying cofactor-1: a change to a cofactor>1 curve would need an explicit subgroup check that is currently absent. A latent, low-severity design assumption to keep in view.

The random oracle is typed, length-framed, and sid-bound

Every EC scheme routes hashing through emp-tool's RO helper (runtime/crypto/ro.h:36-83). The ctor mandates a domain string plus a sid; each field is absorbed as u32_LE(type)‖u32_LE(len)‖bytes with a distinct type id per overload, so the absorb sequence is injective and cross-party byte-exact. squeeze_block truncates the (default) SHA-256 digest to a 128-bit block; squeeze_point is RFC-9380 SSWU_RO hash-to-curve with the domain string as DST. Point wire format is length-prefixed: send_pt writes a u32 length then the 65-byte uncompressed P-256 encoding, and recv_pt bounds the length by MAX_POINT_BYTES before reading — so a malicious point length is bounded, though the count of points is fixed by the caller loop, not the wire.

Production EC scalars come from OpenSSL, not emp's PRG

ECGroup::rand_scalar draws uniformly in [0,order) via OpenSSL BN_rand_range in production (ec.cpp:415); a deterministic emp::PRG path with rejection sampling (ec.cpp:431) is reached only under is_test_mode(), so the reproducible-transcript branch never runs in a release build. Every base-OT scalar (CO's a/bb, PVW's r/s/t/k, CSW's r/α) is thus OpenSSL-random in production.

CRS derivation

PVW and CSW derive public-coin material from the RO rather than trusting a wire value. PVW builds its messy CRS (g0,g1,h0,h1) by hash-to-curve from four fixed labels (pvw.h:42-52); CSW derives T = H1(sid,seed) by squeeze_point (csw.h:118). BMM derives its lattice CRS matrix from sid via SHAKE-256 (§7). All bind sid, so a session id separates transcripts — when a caller sets one (see the CO/PVW zero-sid gap in §11).

3 · CO — Chou–Orlandi "Simplest OT"

base_ot/co.h — 3-move semi-honest OT.

Sender:   a←Z_q ; A=g^a ; send A ; AaInv=(A^a)^{-1}
  per i:  recv B_i ; B_i^a ; BA_i = B_i^a · AaInv
          res[0]=RO("emp-ot:co-base-ot:kdf",sid).absorb(B_i^a).absorb(i) ⊕ data0[i]
          res[1]=RO(…).absorb(BA_i).absorb(i)          ⊕ data1[i] ; send (res0,res1)
Receiver: per i: bb_i←Z_q ; B_i=g^{bb_i} ; if b_i: B_i += A ; send B_i
          As_i = A^{bb_i} = g^{a·bb_i} ; data[i]=RO(…).absorb(As_i).absorb(i) ⊕ res[b_i]

(co.h:18-40 sender, co.h:42-69 receiver.) Correctness: for b=0, Bia = ga·bb = Asi; for b=1, BAi = (gbb·A)a·A-a = ga·bb = Asi. The instance index i is absorbed as a uint64 into each pad, preventing cross-instance pad reuse within a batch; both sides absorb (point, i) in the same order so transcripts agree.

CO's RO is domain- and index-separated, but not session-separated in practice

The RO domain is a single fixed string "emp-ot:co-base-ot:kdf" plus sid.value() (co.h:35, co.h:67). But the standalone test/bench never call set_sid on CO (§11), so it runs under the default zero_block sid — its session binding is effectively absent unless a caller sets one. Streaming: A is sent once, then per-i recv_pt/send_pt, then per-i ciphertexts with a trailing io->flush() (co.h:37-39); two passes over the batch, blocking, no deferred check.

4 · PVW — messy-mode framework OT

base_ot/pvw.h — dual-mode PVW in messy mode: receiver-secure under DDH, sender statistically secure.

The CRS (g0,g1,h0,h1) is hash-to-curve from four fixed labels, so with independent points it is non-DH (messy) (pvw.h:14-24). It is derived lazily on first send/recv via ensure_crs_() binding sid (pvw.h:40-53). The receiver sends (gσr, hσr) per i, keeping all ri live across rounds (pvw.h:90-105). The sender, for b∈{0,1}, draws sb,tb, forms the consistency element ub=gbs·hbt and the masked cb=gis·hit·xb, where xb=gk is a fresh random group element used only as KDF input (pvw.h:60-86). The pad is RO("emp-ot:pvw-base-ot:kdf",sid).absorb(xb).absorb(i); the receiver recovers xσ = cσ − r·uσ and decrypts (pvw.h:119-124).

Malicious security is structural — the ub element is computed but never verified

is_malicious_secure() returns true (pvw.h:28), but there is no runtime consistency check beyond point validation. ub is computed (pvw.h:77) and sent, and the receiver receives both u[bb] but uses only u[σ] in decryption (pvw.h:121) — there is no challenge round, no abort. The guarantees are argued structurally (statistical sender security from the messy CRS; DDH receiver security), not enforced by code; a malicious receiver sending (gi,hi) off the intended DH relation is not detected — messy mode tolerates this by design. An auditor should confirm this against the dual-mode proof. Like CO, the test never sets a non-trivial sid (§11).

5 · CSW — Blazing-Fast FSF-rOT (messy core + deferred check)

base_ot/csw.h — the wired default. A 3-message UC OT under CDH in the observable-RO model; ℓ batched OTs share one (seed,T,z) setup (csw.h:9-48).

Phase A — messy core

CO-style key agreement. Receiver picks seed; both derive T = H1(sid,seed) = RO(to-curve).squeeze_point (csw.h:118, csw.h:192). Receiver sends Bi=gαi·Tbi; sender draws r, sends z=gr, and forms pads pi,0}=H2(sid,i,Bir), pi,1}=H2(sid,i,(Bi/T)r). The Tr term is amortized: ρi,1} = ρi,0} + (−Tr) reuses one negated point per batch (csw.h:117-131). Chosen-input encoding rides in round 2 as ci,b} = pi,b} ⊕ datai,b} (csw.h:136-139), keeping the API at 3 messages; the receiver decrypts from its chosen pad pbi = H2(sid,i,zαi) (csw.h:215-228).

Phase B — deferred extraction check (the malicious teeth)

The sender computes h0i=H3(pi,0}), χi=H3(pi,0}) ⊕ H3(pi,1}), otans=H4(bulk-absorb of all h0), and proof=H3(otans); it sends {χi}, proof (csw.h:156-167). The receiver canonicalizes its pad to the pi,0} slot via χ (otrespi = H3(pbi) ⊕ (bi?χi:0)), recomputes otans', checks H3(otans')==proof, and returns otans' (csw.h:243-256). Four ROs, all binding sid with distinct domains: to-curve, pad (H2), sf-rot-check:h3 (H3), sf-rot-check:agg (H4).

Abort is explicit and via error() — the only base OT with an active check

The receiver aborts on proof != proof_check ("sender misbehavior or selective-failure attack", csw.h:252-253); the sender aborts on otans != otans_prime ("receiver misbehavior", csw.h:171-172). Both use the fatal error() primitive, so they fire under NDEBUG. The aggregate H4 is an exact cross-party byte contract: the sender absorbs h0[] as one framed field (csw.h:161-163) and the receiver must absorb otresp[] byte-identically (csw.h:248-250); any layout divergence yields a hash mismatch and a spurious abort.

A single CSW instance is stateful across the core/check split

send_core/recv_core buffer round-2 bytes without flushing and stash per-instance state (length_, p0_/p1_, p_bi_, b_copy_) as members (csw.h:72-76); *_check runs at session end over that stash. Each *_core call overwrites the members (csw.h:104, csw.h:181), so a single CSW object is not safe for concurrent or interleaved OT batches — an interleaved core call clobbers the prior call's stashed pads before its deferred check runs. b_copy_ copies the receiver's choices so the caller's b[] need not outlive the deferred check. Standalone send/recv just call core+check back-to-back, wire-identical (csw.h:261-268); IKNP/SoftSpoken drive the split via csw_base_.

length≥80 is an assert, elided under NDEBUG

The σ=40 sender-extraction bound (ℓ>2σ, csw.h:46) is enforced only by assert(length >= 80) in both cores (csw.h:105, csw.h:182) — no error()/throw guards it, and length is a runtime int64_t so no static/template guard is possible. CMakeLists.txt forces a Release default (⇒ -DNDEBUG), which elides assert. All shipped callers pass ≥127 (IKNP 127, SoftSpoken n·k≥128, test/bench 128), but a third-party caller invoking CSW::send with, e.g., length=40 in a release build would silently weaken extraction soundness with no abort.

6 · BMM — post-quantum OT over ML-KEM-512

base_ot/bmm.h base_ot/mlkem/bmm.cpp — a 2-message UC OT (2022/415, Protocol 2) over the Regev/Kyber module-LWE PKE at the ML-KEM-512 (KYBER_K=2) parameter set. There is no third-flow extraction check: the receiver freely picks one Kyber public key while the second is fixed by a random-oracle "correlation breaker", which is what makes the reduction tight and removes guess-the-query rewinding. supports_deferred_check() stays false, so an extension drives BMM through the blocking path (bmm.h:60-62).

Protocol flow (roles split across two messages)

Receiver (per OT):  x,e←CBD ; t = A·x + e  (pk=t, sk=x)
                    ĉ_b, c_b ← {0,1}^256 ; r = t − Ĥ_b(ĉ_b) ; c_{1−b} = ĉ_b ⊕ H_b(r,c_b)
                    place c_b,c_{1−b} into wire slots by branch ; send (r, c0, c1)
Sender   (per OT):  parse (r,c0,c1) ; ĉ_0=c1⊕H_0(r,c0) ; ĉ_1=c0⊕H_1(r,c1)
                    per β: pk_β = r + Ĥ_β(ĉ_β) ; ct_β = Enc(pk_β, μ_β)   [μ_β fresh 256-bit]
                            mask_β = H(sid,i,β,μ_β) ⊕ data_β ; send (u_β,v_β,mask_β)
Receiver:           μ_b = Dec(sk_b, ct_b) ; data[i] = H(sid,i,b,μ_b) ⊕ mask_b

(bmm.cpp:144-217 receiver, bmm.cpp:240-308 sender.) Cancellation correctness: for the chosen branch the sender's ĉβ equals the receiver's ĉb (the Hβ XOR unwinds), so pkβ = r + Ĥb(ĉb) = t, the receiver's real key, and Dec(skb,ctb) = μb. For the unchosen branch pk is a hashed-then-shifted value the receiver has no secret key for. The choice bit b never crosses the wire; c0/c1 are symmetric on the wire and the branch assignment (bmm.cpp:180-181) is the only place b enters Round 1, hiding the choice under the RO. The delivered OT message is a 128-bit block K = H(sid,i,β,μ) via squeeze_block (bmm.cpp:109-118), XORed onto data KEM-style — so the base-OT→128-bit-extension contract holds while μ itself is 256-bit.

Wire format

DirectionBytes/OTComposition
recv → send832 B768 B r (uncompressed Rq2, polyvec_tobytes) + 2×32 B correlation strings c0,c1 (bmm.cpp:41)
send → recv1568 B2 branches × 784 B = (640 B compressed u, du=10) + (128 B compressed v, dv=4) + 16 B mask (bmm.cpp:42-43)

The whole batch is serialized into one buffer and transmitted with a single send_data+flush per direction (bmm.cpp:186-187, bmm.cpp:310-311), read with a single recv_data of exactly length·size bytes — no streaming, no length negotiation on the wire.

A doc discrepancy and a non-canonical-r acceptance to note

832 vs 800. The header now states 832 B recv→send (bmm.h:56), but commit 8c6d4d6's message still reads 800 B — d5aed11 widened cβ 16→32 B (the λ=256 fix, +32 B/OT) and updated the header, but the earlier immutable commit message was never reconciled. Non-canonical r. The sender parses r with poly_frombytes, which masks each coefficient to 12 bits (& 0xFFF ⇒ [0,4095], kyber/poly.c:151-158) with no reduction mod q=3329 and no rejection (bmm.cpp:247-248). Correctness survives — polyvec_reduce follows the add (bmm.cpp:263) and r's exact wire bytes are bound verbatim into Hβ — but a malicious receiver gets gratuitous encoding malleability (distinct byte strings for congruent r) that the proof's r∈RqK syntax does not anticipate. Low severity: hygiene, not a break.

7 · BMM — correlation breaker, CRS & oracles

The λ=256 correlation breaker

The symmetric security parameter is λ=256 (kLambdaBytes=32, bmm.cpp:28-31): the correlation strings c0/c1/ĉ and the Hβ outputs are 256-bit via RO::squeeze_digest (bmm.cpp:97-107). This kills the malicious-receiver choice-extraction birthday term: a ~264-query receiver can induce up to q² candidate keys, and q²/2λ = 2128/2256 = 2−128. Commit d5aed11 raised λ from 128 to 256 for exactly this term, at +32 B/OT and no lattice or round change. λ governs only the symmetric side; the lattice stays ML-KEM-512 and the delivered block stays 128-bit.

The 128-bit delivered block vs the 256-bit widening

The λ=256 widening protects only choice extraction. Unchosen-message / output-pad secrecy still reduces to ML-KEM-512 IND-CPA hiding of μ plus the RO pad at the 128-bit block width (bmm.cpp:109-118; squeeze_block truncates SHA-256 to 128 bits at ro.h:54-59). Internally consistent and documented (bmm.h:34-35).

CRS matrix and the three oracle families

The CRS matrix A ∈ Rq2×2 is public-coin from the 16-byte sid: sid → session_seed via SHAKE-256, then A (sender uses AT) via crs_expand_matrix using a sub-seed SHAKE256(session_seed‖0x01) and Kyber's rejection sampler (crs.hpp:84-110). Domain separation is explicit:

  • Ĥβ ({0,1}256→Rq2): rejection-sampled polyvec seeded by SHAKE256(0x02‖sid‖i‖β‖ĉ) (bmm.cpp:74-92). The leading 0x02 tag domain-separates it from the matrix-A expansion's 0x01 (crs.hpp:102) — the explicit hardening added in 8c6d4d6.
  • Hβ (the correlation breaker): RO("emp-ot:bmm:cb", sid).absorb(i,β,r_bytes,c).squeeze_digest (bmm.cpp:97-107); r is absorbed as its exact 768-byte wire serialization so both parties absorb identical bytes.
  • out-key H (μ→{0,1}128): RO("emp-ot:bmm:out-key", sid).absorb(i,β,μ).squeeze_block (bmm.cpp:109-118), per-(i,β) separated.

The two RO-class families use distinct domain strings with type/length-framed absorbs (ro.h:36-83), and the SHAKE-256 Ĥ family is separated by both a different construction and the 0x02 tag — so the oracle families cannot collide across calls. All bind sid and the per-instance index i. Per-OT lattice noise uses fresh independent 32-byte seeds with non-colliding per-component CBD nonces (x@0, e@kK; ρ@0, e1@kK, e2@2kK, bmm.cpp:148-152, bmm.cpp:268-276), so there is no CBD nonce reuse within or across OTs.

8 · Vendored Kyber — provenance & ML-KEM-512 selection

The lattice core under base_ot/mlkem/kyber/ is a subset of pq-crystals/kyber ref, verified byte-identical by diff and SHA-256 for the lifted arithmetic and constant-time primitives: poly.c, polyvec.c, ntt.c, cbd.c, reduce.c, verify.c and their headers — no source-level drift. The post-KyberSlash discipline is intact: poly_compress/poly_tomsg carry the multiply-shift replacements for secret-dividend divisions (kyber/poly.c:33), and verify.c keeps upstream's branch-free verify and the cmov compiler barrier (kyber/verify.c:52).

The six lifted kyber/ headers are pure declaration surfaces, each guarding every symbol behind KYBER_NAMESPACE and pulling its constants from params.h: poly.h declares the poly struct (int16_t coeffs[KYBER_N], an element of R_q) plus its compress / tobytes / frommsg / NTT / reduce / add API (kyber/poly.h:11-13); polyvec.h declares the polyvec = poly vec[KYBER_K] wrapper and the vector NTT / basemul-accumulate (kyber/polyvec.h:9-11); ntt.h declares the zetas[128] twiddle table and ntt/invntt/basemul (kyber/ntt.h:7-17); reduce.h pins the MONT/QINV constants and the two branch-free reductions (kyber/reduce.h:7-14); cbd.h declares the centered-binomial noise samplers poly_cbd_eta1/poly_cbd_eta2 (kyber/cbd.h:8-11); and verify.h declares verify/cmov/cmov_int16 (kyber/verify.h:8-14), of which only cmov_int16 is reached from the BMM path.

Removed vs upstream: indcpa.c, kem.c, fips202.c, symmetric-shake.c, the RNG, and the AVX2 path — BMM calls Kyber's IND-CPA core arithmetic directly, not the CCA/FO KEM wrappers. The gen_matrix rejection sampler (crs.hpp:35-92) and the SHAKE-128/256 surface (symmetric.h:104-198) are reimplemented on OpenSSL EVP rather than lifted — these two files are the primary non-upstream, security-relevant surface and warrant independent audit against FIPS 202 / the Kyber spec (the kyber/ arithmetic stays verbatim).

ML-KEM-512 selection would fail loudly, not silently, on a mis-scoped build

kyber/params.h defaults to KYBER_K=3 (kyber/params.h:4-6), and -DKYBER_K=2 is applied only to the six Kyber .c TUs via scoped COMPILE_DEFINITIONS (CMakeLists.txt:69-70); the C++ side is pinned separately by params.hpp's own #define KYBER_K 2. This is not a silent-miscompile hazard: Kyber's KYBER_NAMESPACE symbol mangling is keyed on KYBER_K (kyber/params.h:9-18), so any .c/.cpp K mismatch becomes a link-time undefined-symbol error, and params.hpp's static_asserts only validate the header-pinned C++ value (always 2). params.hpp also mirrors the resulting constants and warns KYBER_K must never leak transitively through a public base-OT header (bmm.h includes no Kyber headers). The vendored kyber/LICENSE is carried unchanged.

9 · Constant-time posture

The vendored Kyber arithmetic that touches lattice secrets is upstream constant-time: montgomery_reduce/barrett_reduce are branch-free (kyber/reduce.c:16), cbd.c is table-free bit-twiddling, and message embedding uses cmov_int16 rather than a data-dependent branch (poly_frommsg, kyber/poly.c:176-181). Kyber's constant-time equality compare verify() is dead code in the BMM path (there is no re-encryption check — BMM is not a KEM decapsulation); it is only linked because verify.c is in the Kyber source list, and only cmov_int16 is live (kyber/verify.c:16-57).

The glue around the CT core runs secret-choice-dependent control flow

Both base OTs branch on the secret choice bit in the receiver glue, which is not hardened against a local timing/cache adversary (only against the network adversary in the security model): BMM's recv indexes recv_buf by bb·kSendBytesPerBranch and selects c0/c1 pads by a ternary on the choice bit (bmm.cpp:180-181, bmm.cpp:193-197); CSW's recv_core does if (b[i]) B[i] = B[i].add(T) (csw.h:201); and hash_to_polyvec's rejection sampling is variable-time on a secret-derived seed (β=choice is hashed into the seed driving rej_uniform, crs.hpp:35). These are plain array indices and branches, not secret-independent selects — the choice bit's secrecy rests on the RO/algebra argument, and both branches' bytes are on the wire anyway, but the local side-channel posture is weaker than the CT lattice core beneath it. The OpenSSL EVP SHAKE shim is not claimed constant-time, but it processes only public CRS/oracle inputs, not lattice secrets.

Across all four, the only enforced-in-code security check is CSW's Phase-B abort; PVW/BMM's malicious guarantees are argued (structurally / algebraically) and never rejected at runtime, and CO makes no malicious claim. BMM has no protocol-level abort in bmm.cpp: length<=0 is an early no-op, and a malicious peer's malformed input silently produces garbage OT outputs rather than aborting (the only hard aborts are the SHAKE shim's EMP_MLKEM_DIE on OpenSSL allocation failure — a resource abort, not a security check). Its decryption failure is probabilistic per Kyber's noise budget (ML-KEM-512 DFR ≈ 2−139) with no detection or retry; an auditor should confirm that rate is acceptable for the 128-OT bootstrap count.

11 · Test & bench coverage and gaps

test/test_base_ot.cpp sweeps CO / PVW / CSW / BMM at length=128 over loopback, asserting receiver outputs via check_ot (shared PRG seed so both parties draw the same (b0,b1); error() on mismatch, exercising both choice bits across 128 random b). bench/bench_base_ot.cpp mirrors the set and reports µs + bytes each direction; it is a manual harness, not in CI by default. The correctness baseline is honest-execution-only.

Coverage gaps an auditor should weigh

No adversarial/negative test. There is no malicious-sender/receiver case, no test that CSW's send_check/recv_check actually aborts on a tampered proof/otans, no off-curve point-injection test, and for BMM no malformed-r/wrong-c/tampered-mask test — so the malicious-security abort/algebra is unexercised by CI (CSW's csw.h:171-172 / csw.h:252-253 are never hit by the shipped base-OT test). Zero-sid for CO/PVW. The test sets a deterministic sid only on CSW/BMM (test/test_base_ot.cpp:41); CO and PVW run under the default zero_block sid, so their session binding is never exercised with a non-trivial value, and no cross-sid separation test exists. CSW split path. The test drives only the monolithic send/recv; the deferred send_core/send_check split is exercised only indirectly via the IKNP/SoftSpoken extension tests. BMM cross-version SHAKE. On OpenSSL <3.3 the SHAKE-128 squeeze falls back to an O(n²) re-init/re-absorb/FinalXOF loop returning only the new tail (symmetric.h:128-150); byte-identity to the ≥3.3 EVP_DigestSqueeze fast path is untested in-tree (no <3.3 CI matrix).

Threading — two independent hash-context pools, not one

BMM has no internal threading and does one blocking pass per direction, but its thread-safety does not rest solely on the thread_local EVP SHAKE contexts in symmetric.h:152-171: the per-OT h_correlation/out_key helpers construct an emp::RO and dispatch to emp-tool's Hash::hash_once/hash_for_block, a separate SHA-256 (or BLAKE3) mechanism in a different library with its own thread_local context pool. Overall thread-safety spans at least two independent thread-local hash pools in two files. Relatedly, the mlkem keccak_state requires zero-initialization by comment-only contract; the actual hazard from skipping = {} is the garbage opaque EVP-ctx pointer being mistaken for a live context (symmetric.h:73), not the absorbed_len/produced fields (both are overwritten before any read on the sole call path, crs.hpp:57).

Strong correctness baseline

The vendored Kyber arithmetic and constant-time primitives are byte-identical to pq-crystals/kyber ref (diff + SHA-256), carrying the post-KyberSlash multiply-shift discipline and the cmov barrier. All EC schemes route through the length-framed, sid-bound RO and validate every received point (on-curve + non-infinity, fatal). CSW is the only base OT with an active malicious-abort path, and it fires under NDEBUG. Per-OT CBD noise uses fresh seeds with non-colliding nonces; the three BMM oracle families are domain-separated by construction and an explicit 0x01/0x02 tag. The residual audit surface is: the reimplemented OpenSSL-EVP SHAKE/CRS surface (#5), the untested <3.3 SHAKE fallback (#6), the absence of any adversarial test exercising the malicious paths (#7), and the assert-only length≥80 precondition under NDEBUG (#1).
