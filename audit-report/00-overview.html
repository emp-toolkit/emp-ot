emp-ot — repository overview

emp-ot is the oblivious-transfer layer of the emp-toolkit: four base OTs (CO / PVW / CSW / BMM) that bootstrap two OT-extension families — a Random-COT family (IKNP, SoftSpoken, Ferret, SilentFerret, element = block) and a subfield-VOLE family (F2k and Fp, element = an authenticated AuthValue) — all wired to a single streaming begin/next/end skeleton and one abstract OT → COT → RandomCOT hierarchy. It is a primitive library, not a standalone protocol: its outputs (correlated OTs, VOLE correlations) are consumed by downstream 2PC / MPC / ZK backends, and per-protocol malicious security lives in the concrete consistency checks documented in the four detailed reports below.

How this audit was produced

Seven subsystem deep-readers (interfaces/streaming, base-classic, base-PQ, IKNP/SoftSpoken, silent-core, Ferret, sVOLE) read every header and its tests; three independent design critics attacked the merged findings. 114 merged claims were adversarially verified — 96 confirmed, 17 refuted, 1 unverified. Every security-class claim (abort paths, trust flags, delta invariants, domain separation) was verified through two independent lenses. Refuted claims were dropped or corrected in place; the full ledger, the refresh anchor for future re-audits, is audit-report/claims.md.

1 · What emp-ot is

Mechanically the repo is small and almost entirely header-driven: 33 first-party headers (~5,051 lines) plus 11 compiled translation units — five emp bodies (ot_extension/iknp.cpp, ferret/ferret.cpp, ferret/silent_ferret.cpp, softspoken/softspoken.cpp, base_ot/mlkem/bmm.cpp) plus six verbatim pq-crystals Kyber C sources (poly / polyvec / ntt / reduce / cbd / verify). Everything else — the abstract OT hierarchy, all four base OTs, the streaming skeleton, the silent primitives, and both extension families' logic — is consumed inline from headers. The static library emp-ot::emp-ot bundles only the five emp bodies and the Kyber core.

Conceptually it is one abstract interface tower with two concrete protocol families hanging off it:

  • The interface tower — emp-ot/ot.h defines OT → COT → RandomCOT: a chosen-input 1-of-2 OT built generically on top of a correlated-OT primitive via a Davies–Meyer MITCCRH hash (ot.h:45-116), with two declarative capability flags a subclass overrides truthfully (is_malicious_secure(), supports_deferred_check()). emp-ot/ideal.h is a deterministic test double with no security. emp-ot/emp-ot.h is a pure aggregator.
  • The base OTs — emp-ot/base_ot/: the group- and lattice-based 1-of-2 OTs that bootstrap the extensions (CO, PVW, CSW over EC; BMM over ML-KEM-512). CSW is the default base wired into every extension.
  • The shared silent core + streaming skeleton — emp-ot/common/: the cGGM Half-Tree puncturable PRF (cggm.h), the MultiPointGadget that carries the malicious chi-fold check (mp_gadget.h), the primal-LPN amplifier (lpn.h), and the StreamingExtension<Element> lifecycle template (streaming_extension.h) both families inherit.
  • The two extension families — emp-ot/ot_extension/ (RCOT: OTExtension base → IKNP / SoftSpoken / Ferret / SilentFerret, Element = block) and emp-ot/svole/ (one generic Svole<AuthValue> → F2k and Fp, Element = AuthValue).

2 · Architecture — two families over one streaming skeleton

The defining structural fact of emp-ot is that the RCOT family and the sVOLE family are the same machine instantiated on two carriers. Both inherit StreamingExtension<Element> — RCOT with Element = block, sVOLE with Element = AuthValue (a {val, mac} struct carrying its own field ops and chi-fold). Both compose a base-OT bootstrap, a cGGM MultiPointGadget, and an LPN amplifier; they differ only in the carrier's field arithmetic and wire format. The stack below reads bottom-up.

L2base_ot/CO (semi-honest), PVW & CSW (malicious, EC), BMM (malicious/PQ, ML-KEM-512 + vendored Kyber) — the bootstrap OTs

L4common/ (silent core)cggm.h Half-Tree PPRF · mp_gadget.h MultiPointGadget (hosts the App.-C chi-fold check) · lpn.h primal-LPN amplifier — shared by Ferret and sVOLE
L5aot_extension/ (RCOT)OTExtension base (owns the CSW base + Δ) → IKNP (KOS check) · SoftSpoken (subspace-VOLE, sfvole_butterfly.h) · Ferret · SilentFerret — Element = block

The load-bearing invariant an auditor should internalize first

Trust at the interface tower is declarative, not enforced: is_malicious_secure() and supports_deferred_check() are capability flags the base class trusts each subclass to answer truthfully (ot.h:16,23). The streaming layer enforces only a liveness contract (the session tripwire), carrying no cryptographic check itself. All real malicious-security enforcement — the chi-fold consistency checks, the base-OT cross-checks, the PPRF-consistency checks — lives in the concrete subclasses the four detailed reports cover. Reading a trust flag is not reading a proof.

4 · The streaming / consume model

Every extension instance is a single-role streaming producer (the party is fixed at construction). The base StreamingExtension<Element> exposes four subclass virtuals — chunk_size(), begin(), next(out), end() — plus two non-virtual convenience draws the base provides on top (streaming_extension.h:40-131):

run(data, num)     // one-shot: opens+closes a session per call, draining a per-instance leftover buffer
next_n(dst, n)     // buffered incremental draw *within* a caller-owned begin()…end() session

Three access styles, one lifecycle: a fixed-size batch (begin → next* → end), an arbitrary-size incremental draw (begin → next_n → end, refilling a chunk internally), or the library-owned one-shot run. The single leftover buffer lets non-chunk-multiple requests avoid paying a fresh chunk per call, and amortizes the round-end work (refill trees + malicious check) over a whole session rather than per draw. Both the RCOT family (via rcot(data, num) on RandomCOT) and the sVOLE family plug into exactly this template.

The tripwire is liveness, not correctness — and mostly debug-only

The session tripwire has one always-on guard: destructing mid-session calls error() (streaming_extension.h:133-139), because a missed end() desynchronizes the Fiat–Shamir transcript. But enter_session_ / exit_session_ / assert_in_session_ are plain asserts (streaming_extension.h:148-161) — elided under NDEBUG. So double-begin, next-without-begin, and the run()-vs-next_n() mutual-exclusion contract (a shared leftover_ per instance) are enforced only in debug builds; in a release build these degrade to silent draw-ordering corruption. This assert-only posture recurs across the subsystem and is the through-line of the risk register.

5 · Base-OT vs extension crypto worlds

As with emp-tool, it helps to keep two distinct cryptographic regimes apart when reading the code: the public-key base OTs (a handful of group / lattice operations that bootstrap a session) and the symmetric-crypto extensions (millions of correlated outputs from AES/PRG/hash over blocks). They meet only at the OTExtension constructor, which owns a base OT and consumes a few hundred of its outputs to seed the rest.

base_ot/ public-key base OTs

Group-based (CO / PVW / CSW over OpenSSL P-256, with typed length-framed random-oracle hashing and full on-curve point validation in Point::from_bin) and lattice-based (BMM over ML-KEM-512 module-LWE, with the vendored Kyber arithmetic byte-identical to pq-crystals). These produce the small seed of base OTs an extension bootstraps from. CSW is the default; CO/PVW/BMM are live only in the standalone base-OT test/bench.

Report: 1 (Base OT)

ot_extension/ · svole/ symmetric extensions

Millions of correlated OTs / VOLE correlations amplified from that seed using emp-tool's raw-block crypto (CCRH/MITCCRH, PRG, AES, GF(2¹²⁸) f2k) — the Half-Tree cGGM PPRF, the multi-point gadget, and the primal-LPN code. Malicious security here is a chi-fold consistency check seeded from the Fiat–Shamir transcript, not a number-theoretic argument. This is the hot path; the base OT is a one-time cost.

Reports: 2 (OT extension) · 3 (Silent) · 4 (sVOLE)

The wire-trace-hash discipline — a regression guard, not a security property

trace_hash.cpp pins the observed wire bytes of each protocol as a Fiat–Shamir transcript digest (first 16 hex of each direction's SHA-256, ALICE's view). The 18-row baseline table in README.md is the human-maintained source of truth: a changed hash means a wire-format change — fine if intentional, but flag it in the commit. Ferret(b11) and SilentFerret(b11) rows are wire-identical by design (SilentFerret changes only when correction traffic is sent) and must stay equal. This pass independently re-ran EMP_TEST_MODE=1 ./run ./build/trace_hash against a clean HEAD in Release: all 18 rows matched the README baseline current at that HEAD byte-for-byte — no drift. (The baseline has since been regenerated: emp-tool reworked its test-mode seed derivation, which changes every test-mode digest.) Two caveats an auditor should note: the table is Release-only (the RCOT length is (1<<22)+101 under NDEBUG vs (1<<18)+101 in Debug, trace_hash.cpp:121-127, so a Debug run will not match), and it omits SoftSpoken<4> (only <2> and <8> are traced, though the bench sweeps k∈{2,4,8}), so a wire regression specific to the k=4 code path escapes this guard.

Build + tests clean

Both emp-tool and emp-ot build cleanly at -O3 -DNDEBUG -Wall -Werror=switch with zero compiler warnings (Apple clang 21, arm64/NEON), and the full ctest suite passes (11 two-party loopback entries plus the single-process tuning-invariance check).

9 · Security posture at a glance

emp-ot's job is to produce correct correlated randomness fast; the trust model is per-protocol, not uniform. The table below is the first thing an auditor should internalize — which protocol enforces malicious security by what mechanism.

The abort paths are real but never exercised negatively

Every malicious check above aborts via emp-tool's error() (process termination), not a recoverable status. But no test anywhere in the repository drives a deviating party or a malformed message: the malicious=true runs are two honest parties, so the chi-fold / PPRF / CSW / BMM abort branches execute-and-pass, never trigger. A repo-wide search for corrupt/tamper/malform/off-curve/bit-flip fault injection returns zero hits. CI proves honest-execution correctness only — it does not prove any check catches a cheater (risk-register #1). Combined with the NDEBUG-elided misuse asserts, this is the dominant residual audit surface.

Wire format is raw native-endian throughout

Blocks, uint64 counters absorbed into ROs, whole F structs sent via send_data(&x, sizeof(F)), and memcpy'd uint64 indices into SHAKE inputs are all raw host-order bytes with no canonicalization (mp_gadget.h:654, bmm.cpp:82). Consistent with the wider emp stack, but cross-endian peers silently compute divergent transcripts (and fail the malicious checks, or worse in semi-honest mode). No static_assert guards it at this layer.

This overview is the on-ramp; each report below is a full implementation walkthrough with file:line references and a per-report risk register.

Base OTs (classic + PQ)

CO / PVW / CSW over P-256 and BMM over ML-KEM-512: the group- and lattice-based bootstrap OTs, the CSW messy-core + deferred-check split, the typed length-framed RO and point validation, and the vendored-vs-reimplemented Kyber / SHAKE boundary.

The OT/COT/RandomCOT tower and streaming skeleton, IKNP's KOS check, SoftSpoken's subspace-VOLE butterfly and clone_lane COT lanes, the Δ-LSB convention, and the test-coverage gaps in the chosen-input and streaming verbs.

The Half-Tree cGGM PPRF, the multi-point gadget and its App.-C chi-fold, primal-LPN amplification, Ferret's self-bootstrapping rollover, and SilentFerret's prepaid wire-free reschedule (and its PRG-reuse / rollover hazards).

The generic Svole<AuthValue>, the F2k GF(2¹²⁸) and Fp Mersenne carriers, the COPE base sVOLE, the Δ / set_delta invariants, and the val-first AuthValue ABI that downstream emp-zk reinterpret-casts.

11 · Repository caveats an auditor will notice

  • Working notes at the repo root. AGENTS.md and CLAUDE.md are development notes, not part of the library.
  • Pre-alpha API. CMake reads 1.0.0 but names and headers may still move before the first tagged alpha; anything pinned to a release should track v0.3.x instead.
  • The C++17 README floor understates the real requirement. The target exports PUBLIC cxx_std_20 because it follows emp-tool's C++20 contract transitively; a C++17-only consumer will not compile against the installed package.
  • Two nesting levels of emp-ot/. The source path is emp-ot/emp-ot/… (repo dir then include dir), so a repo-relative anchor such as emp-ot/base_ot/csw.h resolves under the inner tree; test files, by contrast, live at test/, not emp-ot/test/.
  • The reorganization is recent. CSW was de-layered and PVWKyber retired in favor of BMM (8c6d4d6); the SoftSpoken clone_lane lanes and the wire-equivalence framework landed in the last handful of commits — read against HEAD @ 481cc1d, not the 0.3.x line.
