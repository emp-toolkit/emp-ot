emp-ot — the sVOLE family

emp-ot's subfield-VOLE extension: a single generic Svole<AuthValue> that produces authenticated VOLE correlations (val, mac) with the invariant macB = macA + Δ·val over two fields — GF(2¹²⁸) (AuthValueF2k, free-XOR + gfmul) and the Mersenne prime field GF(2⁶¹−1) (AuthValueFp). It is the sVOLE analog of Ferret: a base COT bootstrap, a cGGM sibling-OT MPFSS (MultiPointGadget), and an LPN linear-code amplifier, upgraded to malicious security by a Fiat–Shamir chi-fold consistency check that binds the sparse authenticated vector to Δ before amplification.

1 · Role, composition, and party convention

Svole<AuthValue> (svole/svole.h:55-279) is the single sVOLE extension class. It inherits StreamingExtension<AuthValue> and mirrors Ferret's four-step begin/next/end round loop (bootstrap → swap carry → reset tree index → inner begin; rollover check → process one tree; refill → inner end, svole.h:11-34, svole.h:169-194). F2kVOLE and FpVOLE are thin using aliases over the corresponding carrier (f2k_vole.h:143-144, fp_vole.h:199-200).

It composes three unique_ptr members (svole.h:85-88): an inner Ferret (the base COT source), a MultiPointGadget{Sender,Receiver}<AuthValue> (the cGGM MPFSS), and an Lpn<AuthValue,10>. Both gadget sides are always constructed; only one is exercised per party.

outdense sVOLEn = t·2^tree_depth authenticated (val, mac) pairs per round, LPN-densified
L3Lpn<AuthValue,10>public-code linear amplifier; folds the k-element secret into each chunk (svole.h:250-254)
L2MultiPointGadgetcGGM sibling-OT MPFSS → t-sparse authenticated vector + the FTyped chi-fold check
L1Ferret (inner COT)base RCOT source; on F2k its block Δ is the sVOLE Δ

Party convention is carrier-determined. delta_holder_party() returns BOB for F2k (f2k_vole.h:84) and ALICE for F_p (fp_vole.h:101). The load-bearing invariant, stated at svole.h:28-30, is a single chain:

Δ-holder party  ↔  inner-Ferret-ALICE (COT sender)  ↔  MPFSS sender  ↔  rcot caller

The inner Ferret party is set from it in the ctor: inner_party = (party == delta_holder_party()) ? ALICE : BOB (svole.h:95-97). Storage is val-first {val, mac}: the Δ-holder keeps val = 0 throughout (there is no val on the sender side), while the non-Δ-holder has val at the sparse α positions after MPFSS and dense values after LPN (svole.h:32-34, svole.h:246-247).

Round geometry

Two constexpr derive the geometry from PrimalLPNParameter (svole.h:44-49): svole_n = t·2^tree_depth (the trees this round) and svole_M = t + k + 1 (the carry-over: t for MPFSS, k for LPN, and one extra for the chi-fold pseudo-triple). Note this svole_M is not the Ferret-M baked into PrimalLPNParameter (M = k + t·tree_depth + 128, tuning.h:46,53). chunk_size() = 2^tree_depth (svole.h:143-145) and chunk_aligned_buf_sz() = (t − refill_trees)·chunk (svole.h:157-159).

Sizing invariant worth independent confirmation

refill_trees is computed from Ferret-M, not svole_M; the carry-over sizing is correct only if svole_M ≤ Ferret-M holds for every shipped parameter class. It does for all four (b10 66387≤74164, b11 132243≤144070, b12 263665≤280512, b13 526189≤549116), but no assert enforces it (svole.h:152-159). A new parameter class that broke the inequality would silently under-size the carry buffers. See risk #6.

2 · The AuthValue carrier & wire-format traits

The carrier struct is the whole abstraction: it supplies the field type F, {val, mac} storage, field arithmetic (f_zero/f_add/f_sub/f_mul), auth_from_block, the chi-fold helpers (expand_chi, accumulate_VW), the LPN-side ops, and the Svole hooks (delta_holder_party, resolve_delta, on_set_delta, Bootstrap). The contract is documented in common/mp_gadget.h:29-44 and docs/adding-a-protocol.md §1.

Three wire traits

kHasSecretSum, kClearLeafLSB, kChiFoldFlavor. Both sVOLE carriers use the same "Mpsvole" pairing: kHasSecretSum=true, kClearLeafLSB=false, ChiFoldFlavor::FTyped (f2k_vole.h:41-43, fp_vole.h:43-45). The docs warn that only these two established pairings are validated.

AuthValueF2k GF(2¹²⁸)

F = block. f_add/f_sub are XOR; f_mul is gfmul (f2k_vole.h:33-36). Because F₂ is XOR-only, LPN reductions are no-ops and kLpnSafeAddsPerReduce = INT_MAX (f2k_vole.h:71-79). auth_from_block copies the leaf block verbatim as mac with val = 0 (f2k_vole.h:47-49); expand_chi PRG-expands directly into F₂¹²⁸ elements (f2k_vole.h:54-57).

AuthValueFp GF(2⁶¹−1)

F = uint64_t. f_add=add_mod, f_sub = a≥b ? a−b : a+PR−b, f_mul=mult_mod (fp_vole.h:37-40). auth_from_block reduces the low 64 of the leaf block mod p (fp_vole.h:50-54). The struct packs (val_low, mac_high) into one 128-bit block, so auth_add_into does a single _mm_add_epi64 on both lanes and vec_mod reduces both (fp_vole.h:83-96) — relying on the packed layout with no padding.

val-first storage is a load-bearing cross-repo ABI with no compile-time guard

The val-first ordering is a documented ABI contract: downstream emp-zk reinterpret-casts an AuthValue-shaped __uint128_t with val in the low 64 bits and mac in the high 64 (f2k_vole.h:26-31, fp_vole.h:28-31; the consumer casts __uint128_t* to AuthValueFp* on every next_n draw, emp-zk-arith/ostriple.h:59 and ~11 sibling sites). This is enforced only by comments and docs/adding-a-protocol.md ("Don't reorder") — there is no static_assert on sizeof/offsetof anywhere. Reordering the two fields silently corrupts downstream accessors with no build error. See risk #3.

AuthValueFp block-cast is formal alignment UB

AuthValueFp is a plain {uint64_t val; uint64_t mac;} — alignof 8, sizeof 16, no alignas(16) — yet auth_add_into/ auth_partial_reduce reinterpret_cast it to block* and dereference with aligned __m128i semantics (*a = _mm_add_epi64(*a, *b), fp_vole.h:83-96). The stack local AuthValue acc fed in from Lpn::compute_block (common/lpn.h:55-60) is only guaranteed 8-byte alignment. This works today by codegen/allocation accident but is formal UB that can fault under a compiler that emits movaps/movdqa against an 8-mod-16 offset. See risk #2.

3 · MultiPointGadget — cGGM MPFSS + FTyped chi-fold check

The gadget is the shared engine that turns per-tree base COTs into a t-sparse authenticated vector and, in malicious mode, binds it to Δ.

Per-tree sender / receiver

Sender (mp_gadget.h:148-183): fresh cGGM root seed from PRG.random_block, build_sender → leaves + K0, ships the correction c[j] = base_i[j] ^ K0[j], then ships secret_sum = γi − Σ leaf.mac where γi = carry_curr_[tree_idx].mac (svole.h:241-242). Receiver (mp_gadget.h:406-466): recovers α as the MSB-first NOT(LSB) of base_i, receives c[] and secret_sum, sets K_recv[j] = base_i[j] ^ c[j], runs cGGM eval, and α-fills the punctured leaf mac = triple_yz − (secret_sum + Σj≠α mac). It returns rev = bit_reverse(α); the caller writes val there (svole.h:247).

The FTyped chi-fold check

In malicious mode both sides derive a per-tree challenge seed chi_seed = io->get_digest() — bound to the Fiat–Shamir transcript — expand it to leave_n field elements, and accumulate VW[tree] = Σ chi·mac; the receiver additionally records chi_alpha[tree] = chi[rev] (mp_gadget.h:177-182,457-463). At round end (mp_gadget.h:335-347 sender, mp_gadget.h:643-666 receiver) the receiver sends x_star = Σ(β·chi_alpha) + triple_t.val; the sender computes vb = Δ·x_star + triple_t.mac + Σ VW and RO-commits; the receiver recomputes va = triple_t.mac + Σ VW, compares the RO commitments, and aborts error("MultiPointGadget chi-fold check failed") on mismatch. The RO is domain-separated (kDomCheckTyped) and sid-bound (mp_gadget.h:64-66,344,661).

Why the check is sound in this construction

chi_seed comes from io->get_digest(), so a malicious party cannot bias the challenge without desyncing the transcript (FS must be enabled — bootstrap_ calls io->enable_fs, svole.h:203-205). The check binds the sparse vector to Δ before LPN amplification, matching the PCG/Ferret paradigm.

The +1 pseudo-triple is dual-used

Slot t (carry_curr_[param.t]) is both the chi-fold check triple passed to run_end_typed (svole.h:261-264) and the LPN secret's first element (lpn_->compute_slice(dst, carry_curr_.data()+t, chunk), svole.h:250-254). The comment argues both reads see authenticated triples, which "is all the protocol requires," but there is no static_assert or dedicated test for the overlap. Worth independent proof. See risk #5.

The parallel path is closed to sVOLE

mp_gadget.h carries a parallel prepare_all/produce_tree "silent consume" path, but it is static_assert-gated to kHasSecretSum == false (mp_gadget.h:213-214,491-492). Both sVOLE carriers set kHasSecretSum = true, so they cannot use it and run strictly serial — only SilentFerret's AuthValueFerret takes the parallel fast path. There is no ThreadPool/thread/async anywhere in the three svole/ carrier headers.

4 · F₂¹²⁸ bootstrap — Galois packing with optional nesting

AuthValueF2k::Bootstrap::run (f2k_vole.h:104-138) branches on the seed budget: with M = svole_M(param) and the fixed inner class ferret_b10,

if (M > nest_factor(=2) * svole_M(ferret_b10))   // NESTED
     inner Svole(ferret_b10).run(carry_next_, M)   // M seed pairs from a smaller Svole
else                                               // BASE CASE
     pull M*128 raw Ferret COTs, Galois-pack 128 blocks → one mac per seed pair

In the base case the non-Δ-holder extracts the val side from the 128 RCOT LSBs (getLSB → bool_to_block) while the Δ-holder keeps val = 0 (f2k_vole.h:120-137). The nested inner Svole gets a derived child sid and, if Δ-holder, the same Δ (f2k_vole.h:114-117); its own bootstrap always hits the base case, so recursion depth is 2 by construction.

For F2k, the Ferret block Δ IS the sVOLE Δ

resolve_delta returns f->Delta (f2k_vole.h:88) and on_set_delta propagates a user Δ into Ferret via set_delta(bits), asserting LSB == 1 (f2k_vole.h:93-100). That LSB constraint is an assert only — NDEBUG-elidable; the downstream OTExtension::set_delta guard (ot_extension.h:79) is the same elidable assert, and neither path force-sets bit 0. A release build silently accepts an even user-supplied Δ. See risk #4.

The default path is the nested path, but tests only exercise the base case

The default class is b13 (t=1900, k=2¹⁹): svole_M = 526189 > 2·66387, so the shipped F2k default is the nested path. The tests use b11: svole_M = 132243 < 132774, so they exercise only the base-case Galois packing. The nested recursion is triggered by the b13 bench (no correctness oracle) and by trace_hash.cpp's default-param F2kVOLE<> (also b13, a CTest test that only prints a digest) — so it is never correctness-verified. See risk #8.

5 · F_p bootstrap — COPE base sVOLE + pre-stage MPFSS/LPN

AuthValueFp::Bootstrap::run (fp_vole.h:117-195) has three stages and, unlike F2k, no tiered recursion (ferret_b10's n comfortably exceeds the main class's M for b10..b13):

  1. Stage 1 — seed sVOLE. Base_svole (COPE + a chi-fold check) makes 1 + t_pre + k_pre seed pairs (fp_vole.h:123-133).
  2. Stage 2 — base COTs. Pull the pre-MPFSS base COTs from the inner Ferret.
  3. Stage 3 — pre-stage amplify. Run a self-contained mini pre-MPFSS + pre-LPN over n_pre = svole_n(ferret_b10), then memcpy the last M into carry_next_ (fp_vole.h:191-193). This embeds its own gadget + Lpn at ferret_b10 rather than recursing into Svole<>; it is set malicious iff the outer is (fp_vole.h:145-189).

COPE — the CO-backed seed sVOLE

Cope (fp_base_svole.h:30-201) folds MERSENNE_PRIME_EXP = 61 base OTs into an F_p sVOLE. It uses the base OT CO directly — not Ferret — because of a role conflict the header spells out (fp_base_svole.h:16-22): COPE's chooser must be an OT-receiver, but the F_p Δ-holder (ALICE) must be the inner-Ferret COT-sender, so the two cannot share one Ferret. The COPE sender does CO.recv with delta_bool selecting K, seeds m PRGs, computes v = w + (delta_bool ? v : 0) mod p, and prm2pr-packs 61 bit-position shares into one F_p element (fp_base_svole.h:47-119,179-200); the receiver does CO.send both keys and ships tau = w0 + (p − (w1 + u)) (fp_base_svole.h:63-77,141-158).

COPE soundness is ≈ size/2⁶¹, not 2⁻¹²⁸

CO is semi-honest (is_malicious_secure() defaults false, never overridden — ot.h:16, base_ot/co.h:14). The malicious upgrade is a single Wolverine-style chi-fold MAC check per triple_gen (fp_base_svole.h:261-290): the sender ships a mod-p seed, both build chi[i] = seed^(i+1) via uni_hash_coeff_gen, inner-product the macs/x, the receiver sends (xz[0], xz[1]), and the sender checks mac_ip + b == xz[0] + Δ·xz[1], aborting error("base sVOLE check fails") on mismatch. Because the challenge is one uint64 reduced mod p = 2⁶¹−1, the check's statistical soundness is ≈ size/2⁶¹ — tens of bits, not 128. This is inherent to the field, not a bug; noted as risk #9 for the record.

The F_p Δ defaults to zero and nothing forces set_delta

F_p Δ is independent of Ferret's block Δ: resolve_delta returns 0 (fp_vole.h:105) and on_set_delta is a no-op (fp_vole.h:110). The user must call set_delta before begin, but set_delta records no was-set flag and asserts only Δ-holder + !setup_done (svole.h:131-136) — there is no assert or check guarding a forgotten call. A caller who forgets it runs the entire sVOLE with delta_value_ = 0; the chi-fold check is self-consistent at Δ=0 and misses it, and K = mac + Δ·val collapses to K = mac independent of val — trivially forgeable MACs. The test patches Δ from 0→1 to make the honest run pass (test/test_fp_vole.cpp:55-62), which is direct evidence the hazard is real and unguarded. This is the highest-severity item on the page (risk #1).

6 · The fp_utility field kit (GF(2⁶¹−1))

svole/fp_utility.h is the F_p analog of emp-tool's GF(2¹²⁸) helpers. The prime p = 2⁶¹−1 = 2305843009213693951 is defined redundantly as __uint128_t p/pr, block prs/PRs, and uint64_t PR (fp_utility.h:16-24); MERSENNE_PRIME_EXP = 61.

  • Reduction. mod(uint64) is a single Mersenne fold (x&PR) + (x>>61) plus one conditional subtract (fp_utility.h:45-48). The templated mod(k, pv) (fp_utility.h:50-53) handles Cope::prm2pr's up-to-~121-bit __uint128_t input; a single fold reduces to <2⁶² then subtracts, correct because each a[i] < p from a prior mod.
  • Multiply. mult_mod(uint64,uint64) uses mul64 (_mulx_u64 via BMI2 else __uint128_t, fp_utility.h:26-39) then a Mersenne fold; the SIMD mult_mod(block,uint64) reduces both lanes (fp_utility.h:78-98). vec_mod/vec_partial_mod do the block conditional subtract via _mm_cmpgt_epi64.
  • Polynomial hashing. uni_hash_coeff_gen builds chi[i] = seed^(i+1); vector_inn_prdt_sum_red is the mod-p inner product (fp_utility.h:119-139). extract_fp reduces a __uint128_t's low 64 in place (fp_utility.h:115-117).
Single-mod PRG-to-field bias is negligible, not 2⁻³

expand_chi (fp_vole.h:60-64) and the COPE/Base_svole seed draws (fp_base_svole.h:249-250) map a uniform 64-bit PRG word into GF(2⁶¹−1) with a single mod, no rejection sampling. This is sound: 2⁶⁴ = 8·(2⁶¹−1) + 8, so only 8 residues of ~2⁶¹ are over-represented (a 9/8 ratio), giving a statistical distance ≈ 2⁻⁶¹ from uniform — cryptographically negligible. (An earlier reading called this a ~2⁻³ per-element bias; that figure is the pointwise relative excess of those 8 residues, not the distribution's statistical distance. Rejection sampling is correctly omitted.)

7 · Streaming lifecycle, threading, randomness, abort

Lifecycle

(svole.h:169-194) begin() = enter_session_ + lazy bootstrap_ + swap carry + reset tree_idx_ + inner_run_begin_; next() auto-rolls over (end+begin) at tree_idx_ == t − refill_trees then process_one_tree_; end() = run_refill_ + inner_run_end_ (the malicious check) + exit_session_. The one-shot run() and buffered next_n() sit atop this with a per-instance leftover_ buffer, and enter_session_ zeroes leftover_count_ so a fresh begin never serves a stale tail (common/streaming_extension.h:73-154).

Session tripwires — one always-on, the rest debug-only

The destructor's in-session check is an always-on error() (a missed end() desyncs FS), but the no-double-begin / no-end-without-begin tripwires are assert-only and compile out under NDEBUG (streaming_extension.h:133-139).

Session ids & the seed surface

Svole carries its own SessionID (svole.h:73-74) and derives a child sid for the inner Ferret (svole.h:208), the nested Svole (f2k_vole.h:115), and the COPE CO (fp_vole.h:129) — but passes the raw sid.value() to the MPFSS gadget and the pre-stage gadget (svole.h:220-226, fp_vole.h:152-153). The gadgets bind that raw sid into their check RO, so separation is correct only because the pre-stage and main gadgets run sequentially (each absorbing its own transcript) and no concurrent reuse of the raw sid occurs (risk #7). A related non-uniformity: Svole::set_sid is an unguarded plain setter (svole.h:69-74), unlike the OT hierarchy's virtual set_sid which asserts !setup_done (ot_extension.h:102-106); inner_run_begin_ re-reads sid.value() into the gadget every round, so a post-bootstrap set_sid would silently change the domain separator mid-session (risk #8-adjacent, listed as #10-level).

The LPN code is a fixed public matrix across all instances

lpn_->reseed(zero_block) is called with a fixed zero seed, not a sid-derived one (svole.h:108, fp_vole.h:151, common/lpn.h:73) — contrast Ferret, which reseeds from derive_lpn_seed_(sid.value()). Every Svole instance therefore shares the identical public LPN index stream per parameter class. See risk #11.

Randomness & abort

The cGGM per-tree root seed is a fresh PRG.random_block unrelated to Δ (mp_gadget.h:151). All malicious-check and COPE-check failures call error() (process-terminating), not a recoverable return. (Historical note: commit 842da83 converted Base_svole's Cope from raw new/delete to unique_ptr and widened the SoftSpoken OT length to int64_t to fix a silent 32-bit shift overflow, fp_base_svole.h:218,222-232.)

8 · Test & benchmark coverage, and gaps

test/test_f2k_vole.cpp and test/test_fp_vole.cpp drive both the streaming (begin / many-next / end, two rounds) and one-shot (run × kOneshotIters=2) paths at ferret_b11, verifying every output against an authenticated-triple oracle (gfmul(Δ,val)^k for F2k, Δ·val+mac for F_p — test/test_f2k_vole.cpp:11-36, test/test_fp_vole.cpp:22-43). The benches run b13 with no correctness assertion (kOneshotIters=8), reporting per-phase wall time and peak RSS.

The coverage gaps an auditor should weigh

No cheating-party test. The oracle checks honest correctness only; nothing flips a leaf/correction and asserts the chi-fold or COPE check aborts. A regression that silently disabled a check would still pass — the primary gap. Parameter gap. Tests run only b11, exercising the F2k base case but never the nested bootstrap (§4). Δ-forgotten path (F_p). Untested and unguarded (§5). Malicious-only. Both tests use malicious=true over localhost NetIO, so the semi-honest paths that skip run_end_typed are uncovered — though note the base-sVOLE COPE check (Base_svole::sender_check /recver_check) runs unconditionally: Base_svole takes no malicious flag, so it is not among what semi-honest skips.
