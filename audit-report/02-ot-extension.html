emp-ot — OT interfaces & extension

The interface spine every OT protocol plugs into — the abstract OT → COT → RandomCOT hierarchy, the OTIdeal test double, the shared StreamingExtension lifecycle, and the two RandomCOT backends that ride on it: IKNP (classic matrix OT-extension, KOS-hardened) and SoftSpoken (small-field subspace-VOLE with a cGGM puncturable PRF and a recursive butterfly VOLE fold). Plus the wire-trace-hash regression methodology, the malicious-security check machinery, and the clone_lane lane-parallel COT extension. Everything here is a substrate contract: the base classes trust their subclasses, and cryptographic enforcement lives in the concrete protocols.

1 · Role in the OT stack & the two-layer trust model

This subsystem is the interface/streaming skeleton every concrete OT, OT-extension and sVOLE protocol in emp-ot plugs into. emp-ot/ot.h defines the abstract OT → COT → RandomCOT hierarchy; emp-ot/ideal.h a deterministic test double; emp-ot/emp-ot.h a pure aggregator; and emp-ot/common/streaming_extension.h the shared begin/next/end/run/next_n lifecycle that both the RCOT family (OTExtension → IKNP/SoftSpoken/Ferret, Element=block) and the sVOLE family (Svole<AuthValue>) inherit.

RCOTRandomCOTadds the standard 1-bit-per-COT correction on top of a raw rcot() (LSB-of-output convention) so a chosen-correlation COT composes on a random-OT base (ot.h:132-162)
COTCOTgeneric chosen-input send/recv as a MITCCRH Davies–Meyer wrapper over a subclass send_cot/recv_cot (ot.h:39-126)
OTOTabstract 1-of-2 chosen-input OT; is_malicious_secure()/supports_deferred_check() are declarative flags, not enforced (ot.h:9-33)
SEStreamingExtension<E>the 4-pure-virtual chunk lifecycle + non-virtual run/next_n; a session tripwire + an always-on destructor guard, but no crypto (common/streaming_extension.h)
The trust model is split, and the split matters

OT/COT/RandomCOT are abstract contracts with no cryptographic enforcement of their own: is_malicious_secure() and supports_deferred_check() are plain overridable booleans a subclass declares truthfully; the base class trusts the subclass (ot.h:16,23). The streaming layer enforces only a liveness/API-misuse contract (no double-begin, no next without begin, no destruction mid-session) and carries no cryptographic check itself. The real active-security machinery — chi-fold correlation checks, PPRF consistency, base-OT cross-checks — lives entirely in the concrete subclasses covered in §7–§10.

2 · OT / COT / RandomCOT interface hierarchy

emp-ot/ot.h

OT — the abstract chosen-input contract (ot.h:9-33)

Pure virtuals send(data0,data1,length) and recv(data,b,length); is_malicious_secure() and supports_deferred_check() both default false with no runtime verification that a subclass's claim is true (ot.h:16,23). set_sid(SessionID) is non-pure with a default setter that just stores the id (ot.h:29); the comment states callers "are not required to" set it before first use, so a caller that forgets domain separation silently runs with the default zero_block sid — a design choice, not enforced here (contrast OTExtension::set_sid, which asserts !setup_done, ot_extension.h:102-106).

COT — chosen-input over a correlated primitive (ot.h:39-126)

COT implements the generic chosen-input send/recv as a wrapper over a subclass-provided correlated primitive (send_cot/recv_cot, pure virtual, ot.h:42-43). It draws COT outputs, re-randomizes a fresh per-call seed s (prg.random_block + io->send_block(&s,1) on the sender, ot.h:48-50; io->recv_block on the receiver, ot.h:69), seeds a MITCCRH<ot_bsize> via setS(s), and Davies–Meyer AES-hashes data[i] / data[i]^Delta in tiles of ot_bsize = 8 (tuning::cot_chosen_input_tile) before XOR-ing in the chosen messages and shipping 2 blocks/OT (ot.h:57-62). The receiver's recv hashes its own COT output under the same fresh S and selects res[2*j + r[i+j]] — correctness depends on both parties deriving the identical s (one send_block/recv_block round trip) and seeding MITCCRH identically.

MITCCRH ReuseShift defaults to 3 — 8 gids share one key schedule

ot.h:123 instantiates MITCCRH<ot_bsize> supplying only the BatchSize template argument (ot_bsize = 8), so ReuseShift takes its default value 3 from emp-tool/runtime/crypto/mitccrh.h:24. The key schedule is bucketed by gid >> ReuseShift, so 8 consecutive gids hash under one AES key schedule, which the MITCCRH header documents as reducing the multi-instance ideal-cipher bound by 3 bits (125-bit at κ=128) relative to a fresh key per OT (ReuseShift=0). A deliberate throughput/security trade baked into the chosen-input wrapper, unparameterized at the COT layer.

RandomCOT — the ROT-to-COT derandomization (ot.h:132-162)

RandomCOT exposes a single role-implicit rcot(data,num) (ot.h:138) that concrete extensions implement by forwarding to StreamingExtension::run. Its send_cot/recv_cot (ot.h:140-161) add the standard 1-bit correction on top of the raw rcot() output: the sender XORs Delta into the positions the receiver flags over a wire-received byte vector, and the receiver computes bo[i] = b[i] ^ getLSB(data[i]) and ships that correction back over io->send_bool. This is the textbook COT-from-ROT construction and rests on the LSB-of-output convention (LSB(K)=0, LSB(M)=b, LSB(Δ)=1). It costs one bool per OT beyond the RCOT layer's own traffic for every COT consumer.

Peer-supplied choice bytes are sanitized before the table index

The sender's data[i] ^= Delta & tab[bo[i]] indexes a 2-element tab with attacker-controlled wire bytes (ot.h:146-151). This is safe: bo is filled by IOChannel::recv_bool, which bit-unpacks via bits_to_bools whose every path forces the byte to 0/1 (scalar tail & 1; SIMD min_epu8(v,1)). So bo[i] ∈ {0,1} for any wire input and tab[bo[i]] never reads out of bounds. The reinterpret_cast<bool*> over the unsigned char vector at that boundary is a deliberate workaround for std::vector<bool> lacking .data().

There is no abort/error path inside ot.h itself: none of send/recv/ send_rot/recv_rot call error() or throw; length is int64_t with no bounds/overflow check — this layer trusts the caller for lengths and array bounds, and any failure is either a silent logic bug or is caught downstream in the IO layer.

3 · ideal.h — the OTIdeal test double

emp-ot/ideal.h

OTIdeal (ideal.h:6-26) is a COT subclass that fills data from a plain PRG stream — explicitly and only a mock. Its ctor reseeds the PRG with the public constant zero_block "so both parties' OTIdeal PRGs agree without exchanging anything" (ideal.h:9-13): both parties compute the identical deterministic stream with no interaction, which is zero cryptographic security, and the header documents it as "not for production." The header carries no compile-time or runtime guard preventing production instantiation.

OTIdeal has two live correctness bugs, not just a security caveat

(a) Only the first block is filled. Both send_cot and recv_cot call prg.random_block(data) with no length argument (ideal.h:17,21). The signature is random_block(block*, int64_t nblocks = 1) (emp-tool/runtime/crypto/prg.h:186), so for any length > 1 only data[0] is written; data[1..length-1] is whatever uninitialized memory the caller's buffer held, and it differs between the two parties — the mock's COT correlation (and the COT chosen-input wrapper built on it) is silently broken past the first block. (b) Truncating loop + uninitialized Δ. recv_cot iterates for(int i=0; i<length; ++i) — an int over an int64_t length, the one surviving 32-bit truncation in a length path (ideal.h:22). And the ctor sets Delta only when delta != nullptr (ideal.h:12); with the default delta = nullptr, COT::Delta has no initializer (ot.h:40), so each independently-constructed OTIdeal carries its own uninitialized Delta, used inconsistently by the send-side wrapper vs the receiver correction.

Dead API surface

OTIdeal has zero instantiations anywhere in emp-ot's test/ or bench/ trees, nor in the downstream repos (emp-zk, emp-sh2pc, emp-agmpc) — grep finds no OTIdeal callers. Yet it is exported through the umbrella emp-ot.h:5. It implements only send_cot/recv_cot and does not mirror the rcot/begin/ next/end/chunk_size interface that IKNP/SoftSpoken/Ferret actually expose downstream — so its correctness is verified by nothing in CI, which is how the two bugs above survive.

4 · emp-ot.h — umbrella header

emp-ot/emp-ot.h

A pure #include aggregator: ot.h, ideal.h, four base OTs (co/pvw/csw/bmm), iknp, ferret + silent_ferret, softspoken, and both sVOLE flavors (f2k_vole, fp_vole). It defines no symbols and carries no behavior to audit beyond include order (no circular-include risk observed). Because it pulls in every concrete protocol, a consumer that includes only <emp-ot/emp-ot.h> compiles against the full suite; there is no lighter interfaces-only umbrella — a consumer wanting just ot.h + ideal.h (e.g. a mock-only unit test that avoids pulling in base-OT / Ferret machinery) must include those headers directly.

5 · StreamingExtension<Element> — the shared lifecycle contract

emp-ot/common/streaming_extension.h

The template defines the 4-pure-virtual contract chunk_size() / begin() / next() / end() (streaming_extension.h:57,65-67) that every concrete subclass implements, plus two non-virtual public entry points:

  • run(data,num) (streaming_extension.h:73-93) — a one-shot draw that first drains a leftover buffer from a previous run(), loops next() into whole chunks, and for a non-chunk-aligned tail produces one more chunk into leftover_, copies out the requested prefix, and stashes the suffix for the next call. Repeated run(...) calls therefore amortize the chunk cost rather than always rounding up to chunk_size().
  • next_n(dst,n) (streaming_extension.h:110-131) — the incremental-draw counterpart for a caller that owns an open session (external begin() once): it drains leftover, writes whole chunks directly into dst with no intermediate copy, and buffers only a sub-chunk remainder. Per docs/streaming-api.md this exists because per-call begin/end round-end work (refill trees + malicious chi-fold) was previously paid every chunk_size() elements instead of once per session — a documented ~20× slowdown in emp-zk before this API existed. That 20× figure is a doc claim, not reproduced from code in this pass (UNVERIFIED).
The session tripwire is opt-in per subclass — not base-enforced

enter_session_ / exit_session_ / assert_in_session_ (streaming_extension.h:148-161) are protected helpers that subclasses must call themselves from their begin/next/end overrides — there is no NVI dispatch layer, so the base class does not manage the flag automatically. A subclass author who forgets enter_session_() at the top of begin() silently disables the double-begin / next-without-begin protection for that subclass. The current subclasses (IKNP/SoftSpoken/Ferret/Svole) do call them, confirming this is discipline, not enforcement. The three tripwire checks are also plain assert() — compiled out under NDEBUG. And run() vs manual begin/next_n/end mixing is documented as mutually exclusive per instance (streaming-api.md:170-173) but nothing in code enforces it: both share one leftover_ buffer, so mixing them yields silent draw-ordering corruption rather than a crash (see risk #9).

The one always-on safety check

The destructor calls error() unconditionally (not gated by NDEBUG) if the object is destructed while in_session_ is true (streaming_extension.h:133-139), converting a resource-lifetime bug (destroying while the peer expects more data) into a guaranteed abort rather than a silent wire desync. Because end() must virtual-dispatch on a still-alive concrete subclass, the base cannot call end() in its own destructor — so the contract is "caller must end() before the object goes out of scope," enforced by post-hoc detection, not prevention. enter_session_() also resets leftover_count_ = 0 on every begin() so a fresh session never serves a stale tail.

kSenderSendsOnExtend (streaming_extension.h:47-55, static constexpr, default false) is a per-chunk traffic-direction hint for callers multiplexing a streaming extension with a fixed-direction protocol on one socket. It is a static constexpr shadowed per-subclass, not a virtual: read through a StreamingExtension<block>* it is invisible (always the base's false); through a Ferret* it is actively wrong for a SilentFerret object (which shadows it back to false); through a RandomCOT* it does not compile at all (not in RandomCOT's inheritance chain). It is read safely in exactly one place — a concrete OTExtension-typed alias in emp-ag2pc's triple pool — so the trait is real but its only observable in-repo property is the shadowing trap.

6 · OTExtension base — Δ, choice PRG, trust gate

emp-ot/ot_extension/ot_extension.h

OTExtension derives from both RandomCOT and StreamingExtension<block>; each instance is single-role, fixed by party at construction (party == ALICE is the OT sender). Concrete backends implement only begin/next/end; rcot(data,num) forwards to StreamingExtension::run (ot_extension.h:117-119).

  • Trust gate. The shared ctor enforces if (malicious_ && !base_ot->is_malicious_secure()) error(...) (ot_extension.h:152-153) — the only place a malicious extension requires a malicious-secure base OT. This is a precondition, not the security mechanism (see §7/§10).
  • Δ. The sender samples 128 random bits with bit 0 pinned true (the LSB choice invariant), stored as both a Delta block and a delta_bool[128] mirror (ot_extension.h:163-166). set_delta asserts sender role, pre-bootstrap, and bits[0]==true (ot_extension.h:77-79). The whole LSB choice convention and the RCOT correction of §2 depend on this invariant.
  • Choice. The receiver's choice material comes from a single choice_prg (default-seeded from urandom); set_choice_seed reseeds it, asserting receiver role and pre-bootstrap (ot_extension.h:91-95). All three backends derive choice bits from this one PRG.
  • CSW detection. If base_ot->supports_deferred_check() the ctor static_casts it to CSW* (ot_extension.h:154-157), enabling the messy-core-now / extraction-check-deferred overlap; otherwise the blocking base_ot->send/recv path runs. base_check_pending_ tracks that the deferred check runs exactly once in the first end().

7 · IKNP — flow, wire format, KOS chi-fold

emp-ot/ot_extension/iknp.h iknp.cpp

The classic Ishai–Kilian–Nisan–Petrank matrix OT-extension (CRYPTO'03; ALSZ 2013/552), KOS-hardened for malicious security (2015/546). chunk_size() = block_size = tuning::iknp_chunk_ots = 2048 OTs; each next() emits exactly 2048 blocks (16 IKNP rows of 128).

Wire format & the bit-0 encoding

Bootstrap plays 127 base OTs (row 0 is skipped — both sides know it is zero) and reseeds per-row PRGs (iknp.cpp:42-68,149-179). Per chunk, the receiver draws r from choice_prg, computes u_i = G0_i ⊕ G1_i ⊕ r for rows 1..127 and sends them, pins t[0]=r, and transposes via sse_trans_n128 (iknp.cpp:209-228); the sender receives each u_i, sets t row i and XORs u_i when delta_bool[i], forcing row 0 to zero (iknp.cpp:111-126). After the transpose, bit_0(M_k)=choice_k and bit_0(K_k)=0, so Q ⊕ T = r·Δ holds at bit 0 with zero post-processing and the RandomCOT LSB correction lines up. Row 0 is excluded from both wire and FS transcript on both sides, saving 1/128 bandwidth and keeping digests aligned (iknp.cpp:107-110).

The malicious check (corrected)

Malicious mode enables Fiat–Shamir at bootstrap (io->enable_fs(send_first = is_ot_sender()), iknp.cpp:62-63); the u-matrix bytes are auto-absorbed by recv_data/ send_data. The check is not a single chi: per chunk, combine_send/ combine_recv snapshot the evolving FS digest via io->get_digest() after the chunk's u-matrix bytes have crossed and been absorbed (correct commitment-before-challenge ordering, iknp.cpp:134,237), seed a PRG, draw block_size/128 = 16 independent χ coefficients, pack each 128-block group into one F_{2^128} element via GaloisFieldPacking, and fold χ_i·Q_i (sender) or χ_i·T_i and χ_i·R_i (receiver, R_i = r[i]) with vector_inn_prdt_sum_no_red + a final reduce. So the check is a random linear combination of 16 fresh χ per chunk, accumulated across every chunk of the session. The final compare is check_q ⊕ (check_x·Δ) == check_t (iknp.cpp:83-89); mismatch → error("OT Extension check failed"). Soundness is ~2⁻¹²⁸ statistical.

The sacrificial chunk masks the revealed check values, at a cost

send_end_/recv_end_ run one extra full block_size=2048-OT chunk before opening (check_x, check_t), so the revealed values do not determine any user-visible OT (iknp.cpp:70-98,181-201). Only ~128 of those 2048 OTs are semantically needed; the rest are folded harmlessly into the linear check but still cost a full chunk of compute and bandwidth (iknp.cpp:76-79) — a documented efficiency wart, not a correctness/security issue. Semi-honest mode runs none of this: no FS, no chi-fold, no sacrificial chunk.

8 · SoftSpoken — parameters, bootstrap, cGGM

emp-ot/ot_extension/softspoken/softspoken.h softspoken.cpp

Roy's Crypto'22 (2022/192) small-field subspace-VOLE extension, templated on k with n = 128/k sub-VOLEs and Q = 2^k leaves per sub-VOLE. A static_assert requires k ∈ {1..8} and 128 % k == 0, but only {2,4,8} are instantiated (the explicit template-class instantiation list, softspoken.cpp:711-715, plus a k=8 580-block Ferret-bootstrap variant); k=1 is unreachable and the sfvole butterflies static_assert(k >= 2) (sfvole_butterfly.h:305 sender, sfvole_butterfly.h:405 receiver). Chunk sizes: kChunkBlocks = 64 for k=2/4, 128 for k=8; the plane buffer is sized for L1/L2 residency (tuning::softspoken_chunk_blocks, tuning.h:143-156).

  • OT-sender = VOLE/PPRF receiver (bootstrap_send_, softspoken.cpp:121-180): slices Δ into α_i (k-bit field elements), drives cGGM eval_receiver with a bit-reversed path so the storage index equals the field element and leaves[α_i] stays pinned zero_block (the punctured slot). 128 base OTs consumed.
  • OT-receiver = VOLE/PPRF sender (bootstrap_recv_, softspoken.cpp:182-231): builds n Half-Tree cGGM trees, each with a fresh (Δ_tree, root) drawn from choice_prg, shipping per-level K0[h]/K1[h]=K0[h]⊕Δ_tree via base OT.
Choice determinism is load-bearing

Drawing both Δ_tree and root from choice_prg (softspoken.cpp:204-205) makes the receiver's choice-bit stream a pure function of the choice seed — required for set_choice_seed determinism and cross-peer / cross-run consistency (test_choice_seed). The receiver reconstructs all leaves except the punctured α slot, whose pseudorandomness rests on never learning Δ_tree. session_ increments per begin(), giving a fresh AES key domain; the per-chunk PRG offset advances so the keystream across a session's chunks equals one bulk PRG invocation.

9 · SoftSpoken — the subfield-VOLE butterfly

emp-ot/ot_extension/softspoken/sfvole_butterfly.h

The butterfly kernel computes r_x = AES_K(b0+j ⊕ leaves[x]) ⊕ (b0+j ⊕ leaves[x]) (Davies–Meyer under the session key), u = ⊕_x r_x, and the k VOLE planes (sfvole_butterfly.h:22-34). Kernel selection is compile-time: k=8 uses an "oct" kernel fusing rounds 0+1+2 then butterfly_halve<k-3>; k=2/4 use a "quad" kernel fusing rounds 0+1 (k=2 cannot use oct, Q=4<8). Lane width auto-selects AesLane<4> (VAES-512) / <2> (VAES-256) / <1> (AES-NI/NEON) with static_assert(T % L::N == 0); the scalar lane is an always-available fallback, so no intrinsic path lacks one.

The butterfly tile T is now a compile-time LOCAL tuning knob (was a hardcoded T=8)

Both kernels default their j-axis tile width to T = emp::tuning::sfvole_tile<k>() (sfvole_butterfly.h:297 sender, sfvole_butterfly.h:397 receiver), replacing the earlier hardcoded template <int k, int T = 8>. sfvole_tile<k>() reads the per-k macros EMP_TUNE_SFVOLE_TILE_K2/K4/K8 (all default 8, so a stock build is byte-identical to the old default) and is a LOCAL knob: it changes only register/pipeline scheduling, never the wire bytes, and test_tuning_invariance asserts identical output across T ∈ {8,16,32} (tuning.h:118-133). The header notes make tune auto-emits only the k=8 knob — the k≤4 end-to-end optimum is dominated by two-party chunk scheduling a single-process sweep cannot rank, so K2/K4 stay manual.

The punctured leaf's bogus output never reaches an output plane

Security-critical and verified: the receiver substitution reads tweaks[0] = leaves[α] = zero_block (the punctured slot), producing a bogus r_0. Tracing the fold, r_0 is excluded from every w_b plane update and lands only in the A[·][0] accumulator, which butterfly_halve keeps as the even/left operand while v_acc takes only odd R operands — so r_0 ends as u = A[0], which the receiver discards, and never flows into any w_b output plane (sfvole_butterfly.h:393-408,440,481). The plane-major output layout is exactly what sse_trans_n128 consumes, so no gather scratch is needed. One acknowledged inefficiency: sub-VOLE 0 plane 0 is folded then overwritten by the LSB pin (§10) — kept to avoid splitting the sfvole API.

10 · SoftSpoken — COT pipeline & malicious checks

softspoken.cpp

Per-chunk COT pipeline & LSB pinning

The receiver batches all n-1 sub-VOLE differences d_i = u_canonical ⊕ u_temp into one io->send_block, pins plane 0 of sub-VOLE 0 to u_canonical so bit_0(out) carries the choice, and transposes (softspoken.cpp:583-659). The sender receives the batched d_bufs, XORs d_i into w-plane b for each set bit of α_i (derandomization), memsets the mirror LSB pin, and transposes (softspoken.cpp:467-543). The COT relation V ⊕ W = u_canonical·Δ holds at full block level with LSB(K)=0, LSB(M)=u_canonical bit; the RandomCOT wrapper adds the 1-bit correction.

PPRF consistency check (softspoken.cpp:262-346)

Each leaf is expanded by aes_ctr_fill_dm<3> under a fixed key check_K keyed on kPPRFCheckSessionHigh = 0x70505246434B5F00 in the high 64 bits; exp[0] overwrites the real leaf, exp[1:2] is the 2λ check material. The PPRF-sender computes t̂_i and one rolling SHA-256 over all (i,y) and sends t̂ + digest; the PPRF-receiver reconstructs only s'_{α_i} (skipping y == α_i), rehashes, and aborts on mismatch → error("SoftSpoken PPRF check failed").

The check never reveals the punctured leaf

The receiver's reconstruction recovers only the check material s'_{α_i} (PRG′₀), never the real leaf PRG′₁, so leaves[α_i] stays zero_block after the check (softspoken.cpp:323-337). One global rolling digest binds all trees together to catch cross-tree swaps. The check key's high-64 domain (0x70505246434B5F00) is provably disjoint from every per-chunk session key domain (whose high 64 bits are a 16-bit lane_salt, max 0xFFFF), so the two 128-bit AES key domains never collide regardless of the low bits (softspoken.cpp:263,271,480).

Subspace-VOLE chi-fold (softspoken.cpp:673-709)

An IKNP-shape identity Q_i ⊕ T_i = R_i·Δ with R_i = u_canonical[i]: per-chunk d_bufs are FS-absorbed, the χ seed is io->get_digest() (again after commitment), the fold accumulates χ_i·{Q,T,R}_i, and end() runs one sacrificial bs=1 chunk before opening (check_x, check_t); the sender aborts on check_q ⊕ x·Δ != t → error("SoftSpoken subspace VOLE check failed"). FS enablement mirrors IKNP and is guarded by !fs_enabled() so Ferret's nesting is a no-op.

Malicious security is a precondition PLUS three runtime aborts (corrected)

It would be wrong to say the chi-fold and PPRF checks are the sole active-security mechanisms. Malicious security combines (i) the config gate requiring a malicious-secure base OT (ot_extension.h:152-153) with three runtime abort checks all firing via the live-under-NDEBUG error(): (ii) the extension chi-fold (IKNP iknp.cpp:88-89; SoftSpoken subspace softspoken.cpp:450-451), (iii) the SoftSpoken PPRF check (softspoken.cpp:344-345), and (iv) the deferred CSW base-OT extraction check (send/recv_check, driven from the extension end phase, iknp.cpp:95,194 / softspoken.cpp:456,569), which is a distinct check against base-OT selective failure, not part of the chi-fold.

Optional per-phase timers — off by default, zero wire effect

ss_bench_phases.h adds an opt-in profiling facility gated on -DEMP_BENCH_PHASES. When defined it wraps each pipeline stage in a scoped steady_clock accumulator — butterfly, io, derand, transpose, combine, per role — plus an OT counter, and prints a ns/OT breakdown to stderr at each end() (ss_bench_phases.h:25-58). send_begin_/recv_begin_ reset the accumulators (softspoken.cpp:430,551) and send_end_/recv_end_ report them (softspoken.cpp:459,575); the stage scopes sit inside send_chunk_pipeline/recv_chunk_pipeline. Off by default all four macros compile to do{}while(0) (ss_bench_phases.h:60-67), so a production build carries no timers and the wire bytes are unchanged — the instrumentation is pure scope braces plus macro calls, with no reordering of any send_block/recv_block or arithmetic.

11 · clone_lane — lane-parallel COT (commit 481cc1d)

softspoken.h softspoken.cpp ot_extension.h

clone_lane(new_io, lane_salt) builds a fresh SoftSpoken via the CloneInit base ctor — allocating no base OT, sampling no Δ, leaving all mutable streaming/COT state (in_session_, leftover_, mitccrh, prg) at in-class defaults — then value-copies the parent's checked leaves_recv_/leaves_send_, alphas_, Δ, delta_bool, sid, and flips setup_done = true (softspoken.cpp:98-119, ot_extension.h:137-140). A clone must be constructed fresh, never copy-constructed — a genuine copy-construct would alias the parent's leftover_/prg, which the header explicitly forbids (softspoken.h:109-114). Domain separation rides in the high 64 bits of the per-chunk AES session key makeBlock(lane_salt, session); the parent uses salt 0 (byte-identical to pre-lane code).

lane_salt is the sole domain separator, guarded only by a debug assert

clone_lane's security-critical contract — lane_salt nonzero and distinct per lane — is the only thing separating concurrent lanes that share one Δ and one PPRF leaf set. It is enforced by a single assert(lane_salt != 0) that vanishes under NDEBUG (softspoken.cpp:104), and nothing checks for duplicate nonzero salts. A zero or duplicate salt makes makeBlock(lane_salt, session) identical across two lanes that share the same leaves and session counters (both start at 0), so the lanes silently emit byte-identical COT streams — Δ/choice reuse with no runtime detection and no wire-level failure — and the salt is only 16 bits (≤65535 lanes). Each lane's own subspace-VOLE check still passes independently; there is no cross-lane detection.

Lanes skip the PPRF and base-OT checks — they inherit the parent's

A lane clone runs its own per-lane subspace-VOLE check on its own channel but skips the PPRF consistency and deferred base-OT extraction checks entirely (base_ot/csw_base_/ base_check_pending_ null/false, setup_done forced true). All lanes' malicious security therefore transitively depends on the one PPRF check the parent ran at bootstrap over the shared frozen leaves — correct only because the leaves are frozen post-check and value-copied, an invariant guaranteed only by the clone_lane "parent already bootstrapped" assert (softspoken.cpp:103). Note the base-OT extraction check is deferred to the parent's first end() (softspoken.cpp:453-458), so the setup_done assert guarantees only that the PPRF check ran. Parent and every new_io must outlive the clone.

set_sid on a lane clone is a null deref under NDEBUG

OTExtension::set_sid asserts !setup_done then unconditionally dereferences base_ot (ot_extension.h:102-106). A lane clone has base_ot == nullptr and setup_done == true, so under NDEBUG (assert elided) calling set_sid on a lane is a null-pointer dereference instead of a diagnostic. A member of the broader assert-erasure family below.

12 · Wire-trace-hash methodology

docs/wire-trace-hashes.md test/trace_hash.cpp test/trace_equiv.cpp

trace_hash.cpp is a non-cryptographic regression tool: for each protocol it resets a deterministic seed counter, opens a fresh NetIO with Fiat–Shamir enabled, runs the protocol, and prints the first 8 bytes of io->get_send_digest()/get_recv_digest() (trace_hash.cpp:39-49) — pinning exactly the bytes that crossed the wire, from ALICE's view only (ALICE's send digest equals BOB's recv digest). Determinism is entirely contingent on EMP_TEST_MODE=1 — without it every run differs and the tool only prints a stderr warning, it does not abort. Coverage is 18 rows: 4 base OTs, 6 RCOT extensions × {semi, mali}, 2 sVOLE flavors × {semi, mali}.

What the hash table does and does not pin

Cross-protocol invariant: the Ferret/SilentFerret length is chosen as a whole multiple of one SilentFerret round so the two rows are asserted wire-identical by design; a divergence signals a broken equivalence. Release-only baseline: the RCOT length is (1<<22)+101 in Release (to trigger Ferret's per-round rollover and exercise the chi-fold path) vs (1<<18)+101 in Debug, so Release and Debug hashes are not interchangeable. Documented blind spots (wire-trace-hashes.md:107-121): memory-only refactors, non-determinism outside test mode, per-OS/arch entropy, and receiver-side abort paths (a hash captures bytes only up to the abort, so a malicious-check failure mid-protocol is invisible by construction). Notable gap: the table omits SoftSpoken<4> (present in the bench k∈{2,4,8} sweep), so a wire regression specific to k=4 would not be caught. UNVERIFIED: this pass ran no build, so the README baseline table was not regenerated against HEAD.

trace_equiv.cpp is the byte-level companion: it wraps a real NetIO in TraceIO to dump every wire byte to disk per direction per party, but is hardcoded to IKNP semi-honest RCOT only — so localizing a hash divergence in SoftSpoken/Ferret/sVOLE requires ad hoc instrumentation. Its correctness check (verify_rcot) deliberately runs over the raw NetIO, bypassing TraceIO, so the verification round-trip's own bytes do not pollute the trace files.

The abort paths are the least-tested surface

Every test is honest-execution correctness only. No negative/adversarial test feeds a tampered u-matrix, forged d_buf, or corrupted PPRF material to confirm the abort branches actually fire — the chi-fold, PPRF and base-OT check error() paths are unexercised by CI. There is also no test with a non-CSW (blocking-path) base OT, so the base_ot->send/recv branch and the trust-gate error() are untested. Benches (bench.h) do drive begin/next/end directly, but bench code explicitly does not verify correctness — the streaming lifecycle is throughput-tested but correctness-tested nowhere in this repo.

Documentation drift

softspoken.h (softspoken.h:75) and test_softspoken_lanes.cpp (test_softspoken_lanes.cpp:1) cite docs/design/softspoken_lanes.md and threading_model.md, which do not exist in this repo (there is no docs/design/ directory) — the authoritative spec for an upstream API dangles into a consumer repo, against the codebase's own no-downstream-refs convention. The streaming-API doc comment additionally names symbols (sfvole_*_compute_chunk at softspoken.cpp:355, rcot_next at softspoken.cpp:352) that are not the actual function names (the real kernels are sfvole_sender/receiver_butterfly and send/recv_chunk_pipeline).

What is solid

ISA-tier consumption is complete: emp-ot reaches sse_trans_n128 through emp-tool's tier-dispatching transpose.hpp (every tier uses unaligned loads, so 16-byte-aligned buffers suffice), and the sfvole kernels have an always-available scalar lane. Hot-path memory discipline is good — per-chunk state is stack or reused members and BlockVec skips the resize zero-fill. Buffer reuse is correctly guarded against stale-serve/double-consume: enter_session_ zeroes the leftover count, run/next_n drain before extending, SoftSpoken scratch grows monotonically and is fully overwritten each chunk, and SilentFerret snapshots a pristine base0_ to protect its consistency region from the finalize's in-place Δ-XOR.
